# 歌曲下载重做设计

> 日期：2026-04-08
> 执行者：Codex

## 1. 目标

重做现有下载能力，将当前轻量串行下载队列升级为独立下载管理系统，支持：

- 在线歌曲从“更多操作”菜单触发下载
- 点击下载后先选择音质，再加入下载列表
- 独立下载管理页展示任务状态与操作
- 多任务并发下载
- 下载完成后写入音频元数据标签
- 标签写入时优先写入带时间轴歌词，拿不到再降级为纯文本歌词

本次设计只覆盖在线歌曲下载，不覆盖 Local 歌曲导出、批量下载、自定义下载目录和自动扫描入本地库。

## 2. 背景与现状

当前仓库已经存在下载页面、下载控制器和简单任务模型，但设计目标偏轻：

- 任务模型只保存 `id/title/url/status/progress/filePath/errorMessage`
- 下载队列是单进程串行处理
- 下载入口主要挂在播放器页
- 下载完成后不会写入音频元数据标签
- 下载入口和下载页尚未形成统一、完整的产品闭环

同时，仓库里已经存在可复用能力：

- 歌曲更多操作底部菜单：[`lib/shared/widgets/song_actions_sheet.dart`](/Users/wangjian/study/HE-Music-Flutter/lib/shared/widgets/song_actions_sheet.dart)
- 歌曲通用动作分发：[`lib/shared/helpers/detail_song_action_handler.dart`](/Users/wangjian/study/HE-Music-Flutter/lib/shared/helpers/detail_song_action_handler.dart)
- 在线歌曲音质信息：[`lib/features/player/domain/entities/player_quality_option.dart`](/Users/wangjian/study/HE-Music-Flutter/lib/features/player/domain/entities/player_quality_option.dart)
- 在线歌曲 URL 拉取接口：[`lib/features/online/data/online_api_client.dart`](/Users/wangjian/study/HE-Music-Flutter/lib/features/online/data/online_api_client.dart)
- 在线歌词接口与歌词实体：[`lib/features/lyrics/data/datasources/online_lyric_data_source.dart`](/Users/wangjian/study/HE-Music-Flutter/lib/features/lyrics/data/datasources/online_lyric_data_source.dart)

因此本次重做不应继续沿用旧下载模型，而应复用已有歌曲动作入口、音质数据和歌词能力。

## 3. 用户流程

### 3.1 入口规则

- 下载入口只出现在在线歌曲的“更多操作”底部菜单中
- Local 歌曲不显示下载入口
- 下载入口不再依赖播放器页专属按钮作为主入口

### 3.2 主流程

1. 用户在歌曲列表、歌单详情、历史记录、搜索结果等在线歌曲场景中打开“更多操作”
2. 点击“下载”
3. 底部弹出音质选择弹窗，展示可下载音质列表
4. 用户选择某个音质
5. 系统创建下载任务并加入下载队列
6. 下载任务进入独立下载管理页
7. 下载完成后写入标签与歌词
8. 完成后状态更新为“已完成”

### 3.3 音质选择规则

音质选择弹窗必须展示：

- 音质名称
- 格式
- 文件大小（如果接口提供）
- 描述信息（如果平台提供）

默认选中策略：

- 优先使用当前歌曲可用音质中的推荐项
- 推荐项选择逻辑复用当前播放器音质偏好规则
- 如果当前歌曲没有音质列表，则不允许进入下载流程，并提示无法下载

## 4. 设计方案对比

### 方案 A：继续使用 Dio 自管下载队列

优点：

- 现有代码改动最小
- 与当前下载控制器结构接近

缺点：

- 并发、暂停恢复、持久化状态、跨平台一致性需要自行维护
- 后续 macOS 适配会继续堆积复杂度
- 下载管理页和执行内核耦合较重

### 方案 B：使用 `background_downloader` 作为下载执行内核

优点：

- 支持 Android、iOS、Linux、macOS、Windows
- 自带队列、任务状态、并发控制、暂停恢复能力
- 更适合独立下载管理页和后续桌面端

缺点：

- 需要重建当前下载领域模型与执行适配层
- 需要补齐下载任务与插件任务 ID 的映射关系

### 方案 C：按平台拆分实现

优点：

- 可以充分利用平台特性

缺点：

- 维护成本最高
- 平台间行为容易不一致
- 当前仓库阶段不适合引入多套实现

### 推荐

采用方案 B。

原因：

- 当前需求明确包含独立下载管理、并发下载和后续 macOS 方向
- 这些能力正是自管下载最容易失控的部分
- 使用统一下载内核可以把产品逻辑与执行逻辑解耦

参考：

- https://pub.dev/packages/background_downloader
- https://pub.dev/packages/flutter_downloader
- https://pub.dev/packages/audio_metadata_reader

## 5. 架构设计

### 5.1 分层

本次下载重做拆为四个职责单元：

#### A. Download Action

负责从歌曲更多操作菜单发起下载流程：

- 判断当前歌曲是否允许下载
- 拉起音质选择弹窗
- 根据用户选择创建下载任务草稿
- 将任务提交给下载管理器

#### B. Download Manager

负责下载任务生命周期：

- 入队
- 并发调度
- 任务状态同步
- 暂停
- 继续
- 重试
- 删除

#### C. Download Metadata Writer

负责下载后的文件补全：

- 拉取歌词
- 下载或读取封面
- 写入音频标签
- 标记标签写入结果

#### D. Download UI

负责独立下载管理页：

- 分组展示下载任务
- 展示状态、进度、音质、错误
- 提供暂停、继续、重试、删除操作

### 5.2 解耦原则

- UI 层不直接操作下载插件
- 下载执行不依赖具体页面
- 标签写入是下载完成后的独立阶段，不与文件下载阶段混在一起
- 失败状态按阶段细分，避免“下载成功但标签失败”被错误归类

## 6. 数据模型

旧任务模型无法满足新需求，应直接替换。

新任务实体建议至少包含以下字段：

- `taskId`
- `pluginTaskId`
- `songId`
- `platform`
- `title`
- `artist`
- `album`
- `artworkUrl`
- `qualityName`
- `qualityValue`
- `format`
- `sizeBytes`
- `downloadUrl`
- `status`
- `progress`
- `filePath`
- `lyricContent`
- `lyricFormat`
- `tagWriteStatus`
- `errorMessage`
- `createdAt`

其中：

- `taskId` 是应用内部任务 ID
- `pluginTaskId` 是下载插件任务 ID
- `lyricFormat` 区分 `timed/plain/none`
- `tagWriteStatus` 区分 `pending/success/failed`

## 7. 状态机设计

下载任务状态定义为：

- `queued`
- `preparing`
- `downloading`
- `tagging`
- `completed`
- `paused`
- `failed`

状态解释：

- `queued`：已入队，等待调度
- `preparing`：正在准备下载 URL、文件路径或任务注册
- `downloading`：文件下载中
- `tagging`：文件已落盘，正在写入元数据标签
- `completed`：下载与标签写入完成，或下载完成且标签允许降级成功
- `paused`：用户主动暂停
- `failed`：下载阶段失败，或标签阶段失败且需要用户处理

附加状态字段：

- `progress` 只反映文件下载进度
- 标签写入结果由 `tagWriteStatus` 单独表达

## 8. 并发与调度规则

第一版固定最大并发数为 `3`。

调度规则：

- 最多同时执行 3 个下载任务
- 超出并发数的任务保持 `queued`
- `paused` 任务不参与调度
- `failed` 任务只有重试后才重新进入调度
- `tagging` 阶段不占用下载并发配额

这样既能体现下载管理价值，也能避免标签写入阻塞下载吞吐。

## 9. 元数据标签规则

### 9.1 第一版写入字段

- 标题
- 歌手
- 专辑
- 封面
- 歌词

明确不做：

- 音轨号
- 发行年份
- 专辑艺术家
- 评论、流派等扩展字段

### 9.2 歌词写入策略

优先级如下：

1. 优先写入带时间轴歌词文本
2. 如果拿不到带时间轴歌词，则写入纯文本歌词
3. 如果歌词整体获取失败，则允许不写歌词

歌词获取不应阻断文件下载完成，但应影响标签写入结果。

### 9.3 标签失败策略

- 如果文件下载成功，但标签写入失败，任务应保留下载文件路径
- UI 需要能明确区分“文件已下载”和“标签失败”
- 用户应可单独重试标签写入，而不是重新下载整个文件

## 10. 入口设计

共享歌曲更多菜单扩展一个可选下载动作：

- 在线歌曲：显示“下载”
- Local 歌曲：不显示“下载”

该规则应在共享歌曲动作层统一生效，而不是每个页面分别判断。

目标覆盖页面包括但不限于：

- 搜索结果
- 在线评论页歌曲操作
- 我的历史
- 用户歌单详情
- 其他复用共享歌曲动作菜单的在线歌曲场景

## 11. 下载管理页设计

下载管理页为独立页面，不与播放器页绑定。

页面按四个区域展示任务：

- 下载中
- 等待中
- 已完成
- 失败

每个任务项展示：

- 封面
- 标题
- 歌手
- 音质
- 格式
- 文件大小
- 当前进度
- 当前状态
- 错误信息（如有）

可执行操作：

- 暂停
- 继续
- 重试
- 删除

第一版暂不实现批量操作。

## 12. 失败处理

### 12.1 音质解析失败

- 不创建任务
- 直接提示无法下载

### 12.2 下载失败

- 状态为 `failed`
- 保留已选音质信息
- 用户可重试

### 12.3 标签写入失败

- 文件已存在
- 任务保留 `filePath`
- `tagWriteStatus = failed`
- 用户可重试标签写入

### 12.4 歌词获取失败

- 优先降级为纯文本歌词
- 如果纯文本也拿不到，则允许无歌词完成

## 13. 迁移策略

本次为破坏性替换，不兼容旧下载任务模型。

迁移原则：

- 保留“下载管理”产品定位
- 删除旧串行下载控制器的核心逻辑
- 用新的任务实体、任务仓储与下载管理服务替换旧实现
- 原有测试按新交互和新状态机重写

不做旧数据迁移，因为旧模型不具备音质、歌词和标签语义，迁移价值低。

## 14. 测试策略

必须覆盖以下验证点：

### 14.1 交互测试

- 在线歌曲菜单显示下载入口
- Local 歌曲菜单不显示下载入口
- 点击下载后弹出音质选择弹窗
- 选择音质后成功入队

### 14.2 状态机测试

- 并发数生效，超过上限的任务保持等待
- 暂停后任务不再继续推进
- 重试后失败任务重新入队

### 14.3 元数据测试

- 下载完成后进入标签写入阶段
- 优先写入时间轴歌词
- 时间轴歌词缺失时降级为纯文本歌词
- 标签写入失败时状态正确

### 14.4 页面测试

- 下载管理页正确展示下载中、等待中、已完成、失败四组任务
- 任务操作按钮状态与任务状态匹配

## 15. 本次不做

- 批量下载
- 自定义下载目录
- 下载目录设置页
- 自动重新扫描并写入 Local 音乐库
- 音轨号标签
- 下载完成通知细节
- 平台专属差异化下载实现

## 16. 结论

本次下载重做应围绕“在线歌曲下载中心”展开，采用统一下载内核、统一歌曲动作入口和独立下载管理页。

第一版重点不是追求所有下载功能，而是先把以下闭环做完整：

- 从更多操作菜单发起下载
- 音质选择后入队
- 并发下载
- 下载完成后写入封面和歌词标签
- 独立下载管理页可感知并控制整个下载生命周期

这是当前仓库中最适合优先落地的下载产品形态，也是后续扩展批量下载、自定义目录和桌面优化的稳定基础。
