# AGENTS.md

## 项目补充规则

以下规则来自当前项目中已经多次确认过的实现共识，后续默认遵守，除非用户明确要求修改。

### 1. 不要私自脑补

- 不要私自添加身份标签、说明文案或语义强化内容。
- 用户歌单、自建歌单、在线歌单等场景，第一版先按现有已确认页面结构保持一致，不额外加“自建歌单”“我的歌单”等文案。
- 如果用户没有明确要求差异化，先做成和现有已确认实现一致，再继续调整。

### 2. 优先复用现有组件和已确认链路

- 歌曲列表视图默认统一：搜索、首页、歌单详情、专辑详情、歌手详情应尽量复用同一套歌曲 item 与标签计算逻辑。
- 歌单、专辑、视频允许存在两类视图：
  - 搜索页列表视图
  - 首页/广场卡片视图
- “我的收藏歌单”属于在线歌单，样式和交互优先与搜索页歌单 item 保持一致。
- “我的自建歌单”是另一类数据，可以单独处理，但不要影响在线歌单样式。

### 3. 平台列表必须是单一全局状态

- `platforms` 只能维护一份全局状态。
- 首页、排行榜、搜索、广场页等不能各自维护一份平台列表，也不能各自做独立缓存。
- 任何页面发现平台列表未加载时，只能驱动同一份全局 store 去加载。

### 4. 常规接口统一走通用 Dio 拦截链

- 常规业务接口默认使用 `apiDioProvider` 创建的 Dio 实例。
- 这些接口必须统一走已有拦截器链，包括：
  - 登录态注入
  - `401` 登录跳转
  - `403 CAPTCHA_REQUIRED` 验证码流程
  - 通用错误提示
- 只有播放器后台专用链路这类明确隔离的场景，才允许不用这套通用拦截器。

### 5. 详情页改动先保持一致性

- 歌单、专辑、排行榜、歌手等详情页，若结构接近，优先保持头部、播放区、列表区的统一风格。
- 已确认可复用的视觉优化，应同步到同类详情页，而不是只改一个页面。
- 用户歌单详情第一版直接复用在线歌单详情结构，不主动发明新布局。

### 6. 我的页面定位

- “我的页面”只放登录用户相关内容，以及少量高频入口，例如播放历史、本地歌曲、设置。
- 右上角保留设置入口。
- 自建歌单与收藏歌单分开展示，不混为一类。

### 7. UI 默认偏好

- 默认走简洁路线，减少无意义装饰。
- 非必要不要增加毛玻璃、厚重边框、额外说明块、重复 header。
- 如果用户没有明确要求，不要主动给每个页面都加顶部标题栏。
- 组件排版优先稳定、轻量、左对齐，避免因为选中图标、额外标签等导致明显抖动。

### 8. 文案与国际化

- 数量、单位、状态等可见文案优先走标准 i18n，不手写英文后缀。
- 已确认的文案顺序不要随意改，例如歌单数量在前，英文使用 `Tracks`。

### 9. 错误提示规则

- 面向用户的错误提示，优先展示后端返回的业务 `message`。
- 如果后端 `message` 不存在或无法提取，再显示简洁的业务级兜底文案，例如“操作失败”“请求失败”“加载失败”。
- 不要把底层异常细节、网络库错误、堆栈语义或直接的异常对象字符串暴露给用户。
- 全局错误拦截器与页面局部错误提示应尽量复用同一套错误文案解析逻辑，避免同一请求出现两套提示语义。

### 10. 先做一致，再做差异

- 用户先要求“先做成一样”，就按现有页面完整复刻结构与交互，不提前做“我觉得更合理”的差异化设计。
- 差异化能力、扩展字段、额外按钮在用户明确提出后再加。
